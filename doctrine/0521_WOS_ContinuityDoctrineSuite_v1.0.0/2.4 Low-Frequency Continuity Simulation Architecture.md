

### 2.4.1 The Philosophical Verdict: Time is a Universal Constant

If a `Vessel` tracked via live AIS data enters the harbor mouth, or an `Aircraft` enters a high-altitude corridor, it cannot freeze in place the moment the broadcast channel switches focus. If the simulation stops, the _truth_ breaks.

Therefore, when a primitive transitions to a **Dormant State**, it does not halt execution. Instead, the `GridRuntime` or `BroadcastScheduler` drops its execution cadence from the high-frequency render loop (e.g., $60\text{Hz}$) down to a highly optimized, sub-sampled **Macro Clock Baseline** ($1\text{Hz}$, $0.1\text{Hz}$, or bar-synchronized ticks).

Plaintext

```
               [ HIGH-FREQUENCY LOOP (60Hz) ]
               • Active Surface Viewport Focus
               • Full Render, Vector Trail Generation
                              │
                    Broadcast Focus Shifts
                              │
                              ▼
               [ LOW-FREQUENCY CONTINUITY (1Hz) ]
               • Background Matrix Math Only
               • Route Progress / Dead Reckoning
               • Zero Visuals / Zero Event Egress
```

### 2.4.2 Architectural Impact Breakdown

#### 1. Scalability: Vectorized Dead-Reckoning

- **The Concern:** Won't simulating thousands of off-screen elements crash the browser runtime?
    
- **The Solution:** Low-frequency simulation handles **pure matrix math and basic coordinate increments** along a fixed `Route`. It completely bypasses collision detection, physics solvers, UI layout recalculations, and interaction logic. At a $1\text{Hz}$ cadence, modern JavaScript engines can simulate tens of thousands of linear vector updates simultaneously with near-zero CPU footprint.
    

#### 2. Scheduling & Transition Rituals

- **The Benefit:** Because dormant elements are continually creeping forward across the geographic truth in the background, the `BroadcastScheduler` never has to "guess" or randomly spawn entities when the camera cuts to a new zone. The **Transition Ritual** works flawlessly because the incoming active primitives are already exactly where they belong in space and time.
    

#### 3. Future Channel Orchestration (Multi-Surface Synchronization)

- **The Benefit:** Imagine running _Harbor Watch_ on Surface A and _Deep Night Drives_ on Surface B. If a ferry crossing the bay is simulated at a low frequency, it can naturally trigger a camera focus shift on Surface A exactly as it intersects a shipping corridor, completely independent of what Surface B is displaying. The world acts as a single, unified database serving multiple independent broadcast windows.

### 2.4.3 The Continuity Simulation Blueprint

To enforce this without leaking performance, the `Runtime Layer` categorizes dormant background execution into strict sub-sampled profiles:

JavaScript

```
// Conceptual execution routing inside wall/runtimes/GridRuntime.js
function processRuntimeTick(primitive, frameCount) {
  if (primitive.state === 'ACTIVE') {
    updatePhysics(primitive);
    generateVectorTrails(primitive);
    dispatchToEventLayer(primitive); // 60Hz
  } 
  else if (primitive.state === 'DORMANT') {
    // Sub-sample execution: Only tick once every 60 frames (1Hz at 60fps)
    if (frameCount % 60 === 0) {
      incrementRouteDeadReckoning(primitive); // Pure coordinate math
      // NOTE: Authority Dormancy is enforced here.
      // NO event dispatching, NO audio triggers, NO visual updates.
    }
  }
}
```

#### The Sub-Sampling Tiers:

- **Localized Dormancy (e.g., City Vehicles, Subway Trains):** Ticks at **$1\text{Hz}$ (1 update per second)**. Keeps local transit lines and cross-streets perfectly synchronized for sudden camera cuts.
    
- **Macro-Scale Dormancy (e.g., Transatlantic Vessels, Airspace Corridors):** Ticks at **$0.1\text{Hz}$ (1 update every 10 seconds)** or maps directly to incoming AIS data stream intervals. Because the scale of motion is vast and slow, higher granularity is computationally wasteful.
