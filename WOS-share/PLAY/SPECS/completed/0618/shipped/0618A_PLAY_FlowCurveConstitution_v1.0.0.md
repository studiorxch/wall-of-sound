## Purpose

This constitution defines the non-negotiable principles for the Flow Curve Playlist Builder.

The Flow Curve Builder is not a playlist sorter. It is a playlist composition system where the visible curve defines desired movement, and the song pool attempts to satisfy that movement.

The tool exists to help shape long-form playlists and DJ-style mixes using visible, editable energy curves that produce stronger movement than simple BPM/key sorting.

---

## Core Principle

A playlist should move.

The system must support:

- macro arc movement across the full set
- nested local curves inside the larger arc
- up/down and down/up motion
- listener-friendly movement within shorter listening windows
- flexible target durations
- human review and correction
- visible warnings when the library cannot satisfy the intended flow

---

## 1. Flow First

The Flow Curve is the primary authority.

The system must prioritize movement over basic sorting.

A playlist generated by this tool should not simply arrange tracks from slow to fast, fast to slow, or into one giant bell curve. It should create a structured movement path using rises, dips, resets, valleys, waves, ramps, and nested arcs.

### Required Behavior

- The visible curve defines the desired energy path.
- Track placement should attempt to satisfy the curve.
- BPM, Camelot key, duration, and energy are inputs into flow logic, not the whole logic.

### Forbidden Behavior

- Do not reduce the tool to BPM sorting.
- Do not reduce the tool to Camelot sorting.
- Do not treat one full-set bell curve as the default solution for every mix.

---

## 2. Curve as Container

The Flow Curve is a container.

Songs must fit the curve, stretch the curve acceptably, or be rejected as orphans.

The system should try to use all songs from the pool when possible, but it must not force bad fits into the playlist simply to use every track.

### Required Behavior

- Songs that fit cleanly should be placed.
- Songs that stretch the flow should be marked with warnings.
- Songs that break the flow should be moved to an Orphans area.
- Orphaned songs must include clear reasons.

### Example Orphan Reasons

- BPM too low
- BPM too high
- energy too low
- energy too high
- Camelot transition too risky
- no valid slot
- lock conflict

---

## 3. Nested Motion

The default model is one elegant macro DJ arc with nested curves inside it.

The system should support both:

- the full-set narrative arc
- shorter local movements inside the full set

This is required because stream listeners may enter at different moments and may only listen for limited windows.

### Required Behavior

- A full playlist can have a macro arc.
- The macro arc can contain multiple local curves.
- Local curves may include waves, valleys, ramps, pulses, and reset shapes.
- The system must support both up/down and down/up curves.

### Design Intent

The playlist should reward both:

- listeners who start from the beginning
- listeners who join midway

---

## 4. Flexible Duration

The tool must never be hard-coded around 2-hour mixes.

The system must support target durations such as:

- 30 minutes
- 60 minutes
- 90 minutes
- 2 hours
- 3 hours
- 4 hours
- custom durations
- remaining-time rebuilds

### Required Behavior

Curve count and segment length must be derived from:

- target duration
- average track duration
- desired curve density
- selected curve preset
- available track pool

### Forbidden Behavior

- Do not assume 2 hours as a fixed format.
- Do not assume a fixed number of curves.
- Do not assume a fixed number of tracks per curve.

---

## 5. Human Locks Win

Human decisions override algorithmic reordering.

The user must be able to lock important songs so the system reshapes the remaining playlist around those decisions.

### Required Lock Types for Phase 1

- opener lock
- closer lock
- position lock

### Later Lock Types

- segment lock
- neighbor lock
- anchor lock

### Required Behavior

When the curve changes:

- locked songs stay fixed
- unlocked songs may reorder
- warnings update
- orphan status updates

### Forbidden Behavior

- Do not move locked songs unless the user unlocks them.
- Do not silently discard locked songs.
- Do not override locks to improve algorithmic score.

---

## 6. Warnings Teach

Warnings are not just error states. They are library intelligence.

Red and yellow indicators should explain where the song pool cannot satisfy the curve.

### Warning Levels

- **Yellow:** usable but imperfect
- **Red:** likely bad flow or missing requirement

### Required Behavior

Warnings should identify missing zones such as:

- BPM range needed
- Camelot range needed
- energy range needed
- approximate duration needed

### Example Output

Need track around:

- 104–108 BPM
- 7A / 8A / 9A
- energy 0.45–0.55
- duration 4–6 minutes

---

## 7. Planning Before Playback

Phase 1 is a playlist planning tool, not a mixer.

The first build must focus on metadata-driven playlist construction and visual flow editing.

### Phase 1 Includes

- CSV import
- visible Flow Curve
- curve presets
- playlist reordering
- lock support
- orphan handling
- warning indicators
- export to CSV / M3U / JSON

### Phase 1 Excludes

- waveform rendering
- beatgrid analysis
- phrase detection
- transition timing
- live playback control
- OBS overlay
- custom mixer

### Rule

Do not add playback or waveform features until the playlist planning model is stable.

---

## 8. Metadata First

Phase 1 must be built on stable metadata.

### Required Track Fields

- `trackId`
- `title`
- `artist`
- `bpm`
- `camelotKey`
- `durationSeconds`
- `energy`

### Optional Track Fields

- `genre`
- `filePath`
- `sourcePlaylist`

### Deferred Fields

- `mood`
- `density`
- `vocalPresence`
- `waveform`
- `beatGrid`
- `phraseGrid`
- `transitionInPoint`
- `transitionOutPoint`

### Rule

Genre may be used as a filter, but it must not become the primary flow logic in Phase 1.

---

## 9. Song Shape Later

Songs eventually need their own internal flow curves.

A track may contain:

- low intro
- build
- peak
- breakdown
- outro
- density changes
- vocal zones

However, Phase 1 should not depend on full song-curve analysis.

### Phase 1 Rule

Use whole-track metadata first.

### Later Rule

Add song-level curves only after playlist-level Flow Curve logic is stable.

---

## 10. Self-Adjusting, Not Final-Answer Automation

The system should feel like a composition surface, not a one-shot generator.

The user should be able to:

- generate a playlist
- listen to it
- adjust the curve
- lock songs
- remove orphans
- add missing songs
- regenerate the unlocked portions
- export again

### Required Behavior

Curve edits must trigger playlist recalculation.

### Required Constraint

Recalculation should preserve human locks and explain major changes through warnings and orphan reasons.

---

## Phase Boundary Map

| Phase | Focus | Included |
|---|---|---|
| Phase 1 | Flow Curve playlist planning | Metadata import, visible curve, reordering, locks, warnings, orphans, export |
| Phase 2 | Song-level curve model | Intro/mid/outro energy, peak position, basic internal song shapes |
| Phase 3 | Transition map | Waveforms, beatgrid, phrase zones, transition scoring |
| Phase 4 | Playback / stream view | Live visualization, OBS overlay, possible mixer integration |

---

## Build Order Requirement

Implementation must follow this order:

1. Data layer
2. Logic layer
3. Interface layer

### Data Layer First

Define track types, curve types, lock types, project state, warning types, and export formats before UI work.

### Logic Layer Second

Build curve generation, slot generation, scoring, orphan detection, and warning generation before interactive UI polish.

### Interface Layer Third

Build UI around stable data and logic contracts.

---

## Required File Discipline

Do not create God files.

Each module should have one clear responsibility.

Recommended structure:

```text
flow-curve-builder/
  src/
    data/
    logic/
    ui/
```

---

## Success Criteria

Phase 1 is successful when a user can:

1. Import a track pool.
2. Generate a visible Flow Curve.
3. Reorder songs against the curve.
4. Adjust the curve and see the playlist self-adjust.
5. Lock important songs.
6. See red/yellow warnings.
7. Move bad fits into Orphans.
8. Export the final playlist.

The output should be visibly and musically more useful than simple BPM/key sorting.

---

## Final Non-Negotiable

The Flow Curve Builder must remain a playlist composition system.

It must not drift into a generic sorter, a premature mixer, or a waveform editor before the planning model is proven.
