# 1. PURPOSE

This specification defines the official runtime transition doctrine for isolating atmospheric continuity corridors within WOS.

The initial implementation target is:

```
NY Harbor Atmospheric Corridor
```

This spec establishes how WOS transitions from:

- dense urban prototype states
- traffic-oriented runtime behavior
- high-frequency event pressure

into:

- sparse atmospheric continuity
- observability-first pacing
- low-density infrastructural broadcasting

The purpose is NOT:

- deleting systems
- creating separate engines
- branching world simulations

The purpose IS:

```
ONE WORLDMULTIPLE CONTINUITY STATES
```

---

# 2. CONTINUITY ISOLATION DOCTRINE

WOS supports:

```
selective observability
```

rather than:

```
binary world existence
```

The harbor corridor does NOT represent:

- a separate simulation
- a disconnected world
- a paused city

Instead:  
the broadcast frame selectively prioritizes:

- maritime continuity
- atmospheric pacing
- sparse observability
- environmental silence
- symbolic spatial depth

The world persists outside the frame.

Canonical doctrine:

```
The frame of the broadcast is a lens, not a creator.
```

# 3. RUNTIME PARTICIPATION STATES

All runtime systems must support participation gating.

Systems may exist in multiple authority states without:

- despawning
- deletion
- architectural branching

---

## 3.1 ACTIVE

System participates fully in:

- simulation cadence
- rendering
- audio
- event generation
- continuity propagation

Examples:

- active AIS vessels
- harbor atmosphere
- observability camera
- broadcast overlays

---

## 3.2 DORMANT

System preserves continuity identity while reducing authority.

Dormant systems:

- retain state
- preserve continuity
- maintain low-frequency updates
- suppress environmental influence
- reduce event participation

Dormancy is NOT:

- sleep
- pause
- despawn
- deletion

Dormancy IS:

```
continuity-preserving reduced authority
```

Examples:

- distant traffic systems
- inactive district ecologies
- non-observable infrastructure layers

---

## 3.3 SUSPENDED

System temporarily ceases runtime participation while preserving recoverable state.

Suspended systems:

- stop update participation
- stop event generation
- stop rendering authority
- stop audio authority

State persistence remains intact.

Examples:

- traffic AI
- dense urban emitters
- non-relevant world simulations

---

## 3.4 CULLED

System removed entirely from runtime participation.

Culling should remain rare.

Culling is reserved primarily for:

- debug systems
- temporary diagnostics
- non-persistent experimental entities

Canonical WOS doctrine prefers:

```
Dormancy over deletion
```

# 4. LOW-FREQUENCY CONTINUITY SIMULATION

Dormant systems may continue operating at reduced cadence.

Purpose:

- preserve continuity plausibility
- preserve future reactivation integrity
- avoid abrupt world discontinuity
- maintain large-scale persistence illusion

Examples:

- district traffic heartbeat
- distant atmospheric conditions
- regional world timing
- infrastructure state propagation

Recommended baseline cadences:

|State|Suggested Cadence|
|---|---|
|Active|realtime|
|Dormant|reduced cadence|
|Suspended|none|

Exact cadence values remain:

```
implementation baselines
```

NOT immutable doctrine.
