Executive Summary
This specification defines a lightweight water-memory presentation system where vessel wakes briefly accumulate into decaying harbor motion history, allowing ferry corridors, industrial churn zones, and calm protected water to become visually legible without fluid simulation. The core doctrine Ń "Water remembers motion briefly. It does NOT simulate water physics" Ń is correct for WOS atmospheric harbor identity.

Verdict: ACCEPT as canonical-draft Ń well-structured, authority boundaries clear, decay doctrine correct, and ready for REVIEW progression to BUILD.

This is an elegant low-cost solution for making harbor water feel alive without fluid simulation.

Required Amendments (Non-Blocking, Clarifications)
Amendment 1: performance.now() in Reference Logic Ń Determinism Violation
Issue: Reference logic uses performance.now() for timestamp. Water memory is presentation-only, but for replay determinism, should use simulation clock.

Fix:

diff
- createdAtMs: input.nowMs ?? performance.now(),
+ createdAtMs: input.nowMs ?? SBE.Clock.getSimulationTimeMs(),
Add doctrine:

diff
+ ## Determinism Note
+ 
+ Water memory is presentation-only and does not affect runtime truth.
+ However, for consistent replay behavior, timestamps should derive from
+ simulation clock when available. Fallback to wall-clock only in debug mode.
Amendment 2: MAX_ACTIVE_STAMPS = 800 Ń Rationale Missing
Issue: Constant defined but no justification for 800 stamp limit.

Fix:

diff
+ ## Stamp Limit Rationale
+ 
+ MAX_ACTIVE_STAMPS = 800 balances memory accumulation with performance.
+ - Each vessel can contribute multiple stamps per frame
+ - Harbor-scale (200 vessels ? 4 stamps = 800)
+ - Exceeding limit triggers FIFO eviction (oldest stamps first)
+ - Prevents unbounded memory growth
Amendment 3: MAX_ACTIVE_CELLS = 1200 Ń Rationale Missing
Issue: Constant defined but no justification for 1200 cell limit.

Fix:

diff
+ ## Cell Limit Rationale
+ 
+ MAX_ACTIVE_CELLS = 1200 corresponds to:
+ - Screen-space grid: ~40 ? 30 cells = 1200 at full detail
+ - Each cell rendered as one primitive
+ - Target <1ms for cell iteration and decay
+ - Cells beyond limit are merged or discarded
Amendment 4: Cell Quantization Ń Screen-Space Grid Limitations Acknowledged
Issue: Spec acknowledges screen-space grid is camera-local, but doesn't discuss limitations (memory shifts with camera).

Fix: Add limitation note:

diff
+ ## Screen-Space Grid Limitations
+ 
+ Screen-space memory shifts with camera movement.
+ This means water memory is not geographically persistent.
+ 
+ For v1.0.0, this is acceptable because:
+ - Memory is ephemeral (decays in seconds)
+ - Camera movement is relatively slow
+ - The effect is subtle atmospheric residue
+ 
+ Future versions may upgrade to projected water-space cells
+ for geographic memory persistence.
Amendment 5: MIN_STAMP_SPEED_KTS = 0.8 Ń Rationale Missing
Issue: Constant defined but no justification for 0.8 knot threshold.

Fix:

diff
+ ## Minimum Stamp Speed Rationale
+ 
+ MIN_STAMP_SPEED_KTS = 0.8 (~0.4 m/s)
+ 
+ Below this speed, vessel motion is negligible.
+ Stamping would create micro-wakes that clutter water memory
+ without adding meaningful motion history.
+ 
+ Anchored or drifting vessels should not leave significant memory.
Amendment 6: DEFAULT_DECAY_HALF_LIFE_MS = 4500 Ń Rationale Missing
Issue: 4.5 second half-life means memory decays to 25% at 9 seconds, 12.5% at 13.5 seconds. Why?

Fix:

diff
+ ## Decay Half-Life Rationale
+ 
+ 4500ms half-life means:
+ - Memory is visibly present for ~9-13 seconds
+ - Long enough to perceive ferry corridors and churn zones
+ - Short enough to prevent permanent water marks
+ - Aligns with expected viewer attention span for harbor activity
+ 
+ This is not configurable per class in v1.0.0.
+ Future versions may add class-specific half-life modifiers.
Amendment 7: DEFAULT_CELL_SIZE_PX = 32 Ń Rationale Missing
Issue: 32px cell size defined but no justification.

Fix:

diff
+ ## Cell Size Rationale
+ 
+ 32px cells at typical zoom (zoom 12-14) correspond to ~15-30m on water.
+ This resolution is sufficient for:
+ - Ferry lane hints (multiple cells form corridors)
+ - Industrial churn patches (clusters of cells)
+ - Not creating individual cell artifacts
+ 
+ Smaller cells would increase primitive count without visual benefit.
+ Larger cells would lose directional resolution.
Amendment 8: Cleanup Ń Missing Eviction Order for Stamps/Cells
Issue: Exceeding MAX_ACTIVE_STAMPS or MAX_ACTIVE_CELLS triggers eviction, but order not specified.

Fix:

diff
+ ## Eviction Order
+ 
+ When exceeding MAX_ACTIVE_STAMPS:
+ - Evict oldest stamps (by createdAtMs) first
+ - Preserve higher-intensity stamps when possible
+ 
+ When exceeding MAX_ACTIVE_CELLS:
+ - Merge low-intensity cells into neighbors
+ - If merging insufficient, evict lowest-intensity cells first
+ - Preserve lane cells over random churn cells
Amendment 9: clearWaterMemory() Ń Missing Effect on Active Stamps
Issue: Function signature listed but no specification of whether it clears both cells AND pending stamps.

Fix:

diff
+ ## clearWaterMemory() Behavior
+ 
+ clearWaterMemory() must:
+ - Remove all WaterMemoryCells from the grid
+ - Discard all pending stamps
+ - Reset all accumulators to zero
+ - Preserve runtime flags (showMaritimeWaterMemory, etc.)
+ - Not affect WakeAuthority or vessel truth
+ 
+ After clear, memory accumulation restarts from empty.

Final Statement
v1.0.0 is a canonical water-memory presentation specification for WOS maritime systems. It defines a lightweight, decaying environmental history system where vessel wakes briefly accumulate into harbor motion memory Ń ferry corridors, industrial churn zones, cargo trails, tug agitation, and calm protected water become visually legible without fluid simulation. Water memory is presentation-only: it may imply recent motion but never invents current entities. All memory decays, no permanent marks persist. Integration with 0526A WakeSignature is optional and graceful. Performance targets are bounded (800 stamps, 1200 cells, screen-space grid). The spec is ready for REVIEW progression to BUILD.
