Executive Summary
This specification defines the runtime visibility-class resolver that converts AtmosphericReadability outputs, density pressure, zoom context, surface preset context, and environmental presentation state into bounded visibilityClass tokens consumed by MapStyleAuthority and renderers. It enforces downward-only visibility tightening (never elevation), preserves the supremacy of AtmosphericReadability as the upstream atmospheric truth source, and maintains strict separation from runtime truth.

Verdict: ACCEPT as canonical-draft ﾑ constitutionally sound, authority boundaries clear, downward-only rule correct, and ready for REVIEW progression to BUILD.

This is a model interpretation-layer specification that closes the visibilityClass: null gap in the presentation manifest.

Critical Strengths (Preserve)
1. Core Doctrine: "Visibility classes constrain presentation. They do NOT create simulation truth."
This is the constitutional foundation. Preserve verbatim.

2. Downward-Only Visibility Rule
"Visibility may only tighten. Once an upstream authority has constrained a vessel, downstream systems may not elevate it."

Canonical order of restrictiveness:

text
FULL ? REDUCED ? SILHOUETTE ? MARKER_ONLY ? LIGHT_ONLY ? ATMOSPHERIC_HIDDEN
A resolver may move rightward. It may not move leftward.

Example: AtmosphericReadability says MARKER_ONLY ? Surface preset may tighten to LIGHT_ONLY ? Renderer may not restore SILHOUETTE or FULL.

3. Clear Architectural Placement
text
AISRuntime ? ContinuityEngine ? AtmosphericReadability ? ContinuityDensity ? SurfaceStylePresetRuntime ? VisibilityClassRuntime ? MapStyleAuthority ? Manifest ? Renderer
VisibilityClassRuntime sits after runtime truth and before manifest consumption ﾑ correct placement.

4. Visibility Tightening Sequence
Reference logic applies tightening in order:

Start with upstream visibilityClass (AtmosphericReadability) or FULL

Apply density tightening (ｳ0.75 ? MARKER_ONLY, ｳ0.45 ? REDUCED)

Apply distance/zoom tightening (ｳ8km ? LIGHT_ONLY, ｲ10.5 zoom ? MARKER_ONLY)

Apply preset tightening (SIGNAL_DRIFT ? SILHOUETTE, BROADCAST_FAILURE ? LIGHT_ONLY)

Apply population tier tightening (GHOST ? LIGHT_ONLY)

Enforce downward-only rule

5. Strict Separation from Runtime Truth
VisibilityClassRuntime may observe:

AtmosphericReadability outputs

ContinuityDensity clutter pressure

PopulationHierarchy tier

vessel distance, zoom level

Surface preset context

May NOT mutate: AISRuntime, vessel coordinates, heading, speed, wake buffers, population tier, camera routing, overlay semantics.

6. Population Tier Integration
GHOST tier tightens to LIGHT_ONLY ﾑ prevents ghost vessels from receiving full detail while preserving subtle atmospheric presence.

7. Debug API Surface
_wos.visibilityClass.resolve(), batch(), snapshot(), matrix(), testPreset(), testDistance(), testDensity(), constants() ﾑ comprehensive debugging.

8. Integration Contracts
MapStyleAuthority: consumes resolved class in generateManifest()

SurfaceStylePresets: may tighten visibility, never reveal hidden vessels

MaritimeStyleRegistry: consumes class via applyVisibilityClassToStyle()

9. Source Authority Tracking
sourceAuthority field identifies which system caused the final class: ATMOSPHERIC_READABILITY, DENSITY, ZOOM, PRESET, or FALLBACK. Critical for debugging.

10. tightenedFrom Field
Tracks original class before tightening. Enables debugging of multi-stage visibility reduction.

Required Amendments (Non-Blocking, Clarifications)
Amendment 1: Missing Justification for Density Thresholds (0.45, 0.75)
Issue: Thresholds MEDIUM_DENSITY_THRESHOLD = 0.45 and HIGH_DENSITY_THRESHOLD = 0.75 appear without rationale.

Fix: Add justification:

diff
+ ## Density Threshold Justification
+ 
+ MEDIUM_DENSITY_THRESHOLD (0.45): At this density level,
+ clutter starts to affect readability. Vessels should transition
+ from FULL to REDUCED to maintain visual clarity.
+ 
+ HIGH_DENSITY_THRESHOLD (0.75): At this density level,
+ the harbor is saturated. Most vessels should reduce to
+ MARKER_ONLY to prevent cognitive overload.
+ 
+ These thresholds align with ContinuityDensity's density class
+ boundaries (SPARSE <0.25, ACTIVE <0.55, DENSE <0.78, SATURATED ｳ0.78).
+ 
+ 0.45 is approximately the midpoint between ACTIVE (0.25) and DENSE (0.55).
+ 0.75 is near the SATURATED threshold (0.78).
Amendment 2: Missing Justification for Zoom Thresholds (10.5, 12.0)
Issue: LOW_ZOOM_THRESHOLD = 10.5 and MID_ZOOM_THRESHOLD = 12.0 appear without rationale or relationship to Mapbox zoom levels.

Fix: Add justification:

diff
+ ## Zoom Threshold Justification
+ 
+ Mapbox zoom levels in WOS maritime context:
+ - Zoom 8-10: Harbor overview, vessels appear as small markers
+ - Zoom 10-12: District-level view, vessel shapes become distinguishable
+ - Zoom 12-14: Detailed harbor view, vessel classes readable
+ - Zoom 14+: Pier-level detail
+ 
+ LOW_ZOOM_THRESHOLD (10.5): Below this zoom, vessel hull detail is
+ not visually useful. MARKER_ONLY reduces visual clutter.
+ 
+ MID_ZOOM_THRESHOLD (12.0): Above this zoom, vessels can be rendered
+ with class-specific detail (subject to other constraints).
+ 
+ These thresholds are advisory defaults. Surface presets may override.
Amendment 3: Missing Justification for Distance Thresholds (3.0km, 8.0km)
Issue: MID_DISTANCE_KM = 3.0 and FAR_DISTANCE_KM = 8.0 appear without rationale.

Fix: Add justification:

diff
+ ## Distance Threshold Justification
+ 
+ MID_DISTANCE_KM (3.0km): At this distance, vessel class is
+ still readable but fine details fade. REDUCED visibility is appropriate.
+ 
+ FAR_DISTANCE_KM (8.0km): Beyond this distance, hull detail is
+ no longer visible. LIGHT_ONLY preserves atmospheric harbor presence
+ without attempting impossible detail rendering.
+ 
+ These thresholds are based on typical human visual acuity for
+ vessel-sized objects over water. Fog and low-light conditions
+ are handled upstream by AtmosphericReadability.
Amendment 4: resolveBatch ﾑ Missing Performance Guidance
Issue: Batch resolution is mentioned but no guidance on performance expectations or when to use batch vs individual resolution.

Fix: Add performance guidance:

diff
+ ## Batch Resolution Performance
+ 
+ `resolveBatch` should process all inputs in O(n) time.
+ 
+ Recommended usage:
+ - Call once per frame for all visible vessels
+ - Cache results per entity until input factors change
+ - Maximum batch size: 500 vessels (harbor-scale)
+ - Target latency: < 5ms for full batch
+ 
+ Individual `resolveVisibilityClass` calls are acceptable for
+ debugging or single-entity inspection but should not replace
+ batch processing in production render loops.
Amendment 5: Missing Population Tier Constants
Issue: Reference logic uses input.populationTier === "GHOST" but no constants for tier thresholds. Other tiers (HERO, MID, BACKGROUND) are not mentioned in tightening logic.

Fix: Add tier tightening guidance:

diff
+ ## Population Tier Visibility Influence
+ 
+ HERO: No tightening (remains at upstream visibility)
+ MID: No tightening
+ BACKGROUND: May tighten by one step (e.g., FULL ? REDUCED) under high density
+ GHOST: Tightens to LIGHT_ONLY (as specified)
+ 
+ This ensures hero and mid-tier vessels remain visible for cinematic framing,
+ while background and ghost vessels suppress appropriately.
+ 
+ Implementation of BACKGROUND tightening is deferred to v1.1.0.
+ v1.0.0 only implements GHOST tightening.
Amendment 6: Missing entityType: "INFRASTRUCTURE" | "UNKNOWN" Usage
Issue: Input includes entityType but reference logic never uses it. Is infrastructure handled differently?

diff

+

+ Future versions may differentiate:
+ - INFRASTRUCTURE (piers, bridges, buoys): May have different visibility rules
+ - UNKNOWN: Conservative fallback (MARKER_ONLY at distance)
+ 
+ For v1.0.0, `entityType` is reserved for future expansion.
+ Implementations may ignore it or log warnings for unexpected values.
