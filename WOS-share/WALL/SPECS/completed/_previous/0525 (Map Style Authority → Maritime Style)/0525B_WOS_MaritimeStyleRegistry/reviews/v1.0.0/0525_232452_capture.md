Strengths (Preserve)
1. Canonical Distinction: unknown vs default
"unknown = recognized taxonomy class. default = defensive fallback for invalid, missing, future, or unmapped class keys."

This prevents semantic conflation and ensures safe resolution for all vessels.

2. Core Principle: "Far Vessels Are Atmospheric Infrastructure"
"Distant vessels should behave visually like small living harbor lights, not miniature tactical entities."

This is the correct philosophical foundation for far-light behavior. Preserve verbatim.

3. Far-Light Doctrine
"Far lights should be alive but not loud. Far-light behavior may include low alpha, subtle halo, low-frequency twinkle, class-tinted glow, distance-weighted softness, occasional micro-pulse, weather-dampened intensity. May NOT include attention-grabbing flashing, tactical target glow, high-frequency blinking, aggressive selection rings, label-first rendering, or gameplay enemy-marker behavior."

This is critical for maintaining atmospheric harbor realism without fatigue.

4. Visibility Class Response Table
VisibilityClass	Allowed Presentation
FULL	detailed silhouette, colors, wake, hover
REDUCED	simplified silhouette, softer wake
SILHOUETTE	class-shaped outline, minimal fill
MARKER_ONLY	small marker, class-safe tint
LIGHT_ONLY	far-light point, halo, twinkle, no hull
ATMOSPHERIC_HIDDEN	no rendering
Clean separation: MaritimeStyleRegistry responds to visibilityClass, never assigns it.

5. Hover Card Separation
"MaritimeStyleRegistry owns hover-card visual treatment only. OverlayGrammar or a future hover-card semantic spec owns what fields appear, field order, semantic grouping, label names, telemetry hierarchy."

Critical separation between visual treatment and semantic content.

6. Density Response
applyDensityPressureToStyle suppresses visual load only Ń never mutates runtime state. Far-light alpha, wake alpha, and glow strength are reduced proportionally to clutter pressure.

7. Complete Default Registry
All 11 canonical classes plus default have defined styles with class-specific variations:

Tanker: warm red/orange accents

Ferry: civic blue, higher far-light alpha

Recreational: smaller scale, higher twinkle strength

Military: subdued, low far-light, low glow

Industrial: orange/utility-coded

8. InterpolationCurve Enum
Includes CUBIC_GLIDE as default Ń appropriate for smooth visual motion without runtime authority leakage.

9. Execution Flow
text
runtime truth ? interpretation ? presentation
NEVER: presentation style ? runtime mutation
10. Debug APIs
_wos.maritimeStyle.palette(), validate(), inspectClass(), compareClasses(), visibilityMatrix(), densityTest() Ń comprehensive debug surface.

Required Amendments (Non-Blocking, Clarifications)
Amendment 1: twinkleRateHz Ń Missing Rationale
Issue: twinkleRateHz default is 0.18 Hz (once every ~5.5 seconds). No justification for this specific rate.

Fix: Add rationale:

diff
+ ## Twinkle Rate Rationale
+ 
+ 0.18 Hz (~5.5 seconds per twinkle) is intentionally slow.
+ Faster rates (ł 1 Hz) create attention-grabbing blinking
+ that breaks atmospheric calm.
+ 
+ The goal is subtle "alive but not loud" behavior.
+ Twinkle should be noticed only when the viewer looks directly.
Amendment 2: MAX_HOVER_HOLD_MS = 3200 Ń Missing Rationale
Issue: Maximum hover hold is 3.2 seconds. Why this bound?

Fix: Add rationale:

diff
+ ## Hover Hold Maximum Rationale
+ 
+ 3200ms is the upper bound for hover persistence.
+ Longer holds risk blocking the view and creating UI fatigue.
+ 
+ Default hold is 1400ms Ń long enough for slow atmospheric reading,
+ short enough to fade when the viewer looks away.
Amendment 3: DEFAULT_DENSITY_SUPPRESSION_STRENGTH = 0.50 Ń Missing Interaction with labelVisualSuppressionStrength
Issue: DEFAULT_DENSITY_SUPPRESSION_STRENGTH = 0.50 appears as a constant but labelVisualSuppressionStrength in DensityResponseStyle is set to 0.75 in the default style. No explanation of why label suppression is stronger than general clutter suppression.

Fix: Add clarification:

diff
+ ## Density Suppression Strength Clarification
+ 
+ `labelVisualSuppressionStrength` (0.75) is higher than
+ `clutterSuppressionStrength` (0.50) because labels contribute
+ disproportionately to visual clutter.
+ 
+ Under high density pressure, labels should be suppressed earlier
+ than hull or wake rendering to maintain readability.
+ 
+ This does NOT affect label eligibility Ń only visual presentation
+ of labels when they are already eligible.
Amendment 4: compactScaleMultiplier vs detailedScaleMultiplier Ń Missing LOD Integration
Issue: Both multipliers are defined but no guidance on when each is used (zoom level? density? population tier?).

Fix: Add LOD guidance:

diff
+ ## Scale Multiplier Usage
+ 
+ `compactScaleMultiplier` is used when:
+ - Zoom level is distant (low-detail mode)
+ - High density pressure is active
+ - Population tier is BACKGROUND or GHOST
+ 
+ `detailedScaleMultiplier` is used when:
+ - Zoom level is close (high-detail mode)
+ - Vessel is HERO or MID tier
+ - AtmosphericReadability returns FULL visibility
+ 
+ The renderer chooses which multiplier to apply based on
+ viewport scale and vessel observability context.
+ 
+ MaritimeStyleRegistry does not make LOD decisions.
Amendment 5: applyVisibilityClassToStyle Ń Missing Handling for REDUCED Marker Radius
Issue: REDUCED reduces stroke width but does not adjust marker radius. For MARKER_ONLY and LIGHT_ONLY, marker radius is set to 0.0. For REDUCED, marker might still be full size.

Amendment 6: CUBIC_GLIDE Interpolation Curve Ń Missing Definition
Issue: CUBIC_GLIDE is referenced in the enum and default style but not defined.

Fix: Add definition:

diff
+ ## CUBIC_GLIDE Definition
+ 
+ `CUBIC_GLIDE` is a cubic ease-in-out curve with a slower start and end:
+ 
+ t' = t < 0.5 ? 4 * t^3 : 1 - pow(-2 * t + 2, 3) / 2
+ 
+ This produces smooth visual motion with gentle acceleration and deceleration,
+ appropriate for maritime vessels that should feel heavy and continuous.
+ 
+ Linear interpolation is forbidden for motion presentation as it creates
+ mechanical, teleport-like visual artifacts.
