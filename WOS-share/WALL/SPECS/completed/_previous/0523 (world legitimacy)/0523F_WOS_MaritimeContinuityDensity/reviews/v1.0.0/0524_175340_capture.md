Executive Summary
This specification defines continuity density as a deterministic interpretation layer Ń it observes runtime state and computes density pressure descriptors, but never mutates AIS truth, vessel lifecycle, population tiers, wake authority, or renderer state. The core doctrine "ContinuityDensity may describe pressure. ContinuityDensity may never resolve pressure by mutating truth" is constitutionally sound.

Verdict: ACCEPT as canonical-draft Ń well-structured, authority boundaries clear, deterministic requirements explicit, and ready for REVIEW progression to BUILD.

This is a model interpretation-layer specification for density monitoring.

Strengths (Preserve)
1. Core Doctrine: "May describe pressure. May never resolve pressure."
This prevents density from becoming a hidden correction authority. Preserve verbatim.

2. Clear Output Hierarchy
Global scores (globalDensityScore, globalContinuityLoad, globalClutterPressure) + sector-level detail + density classes (SPARSE, ACTIVE, DENSE, SATURATED).

3. Relationship to AtmosphericReadability
ContinuityDensity provides clutterPressure to AtmosphericReadability, which consumes it for label suppression and readability reduction. This is the primary downstream consumer Ń clean interface.

4. AIS Supremacy Rule
"AIS vessels must not be suppressed by ContinuityDensity. High AISPressure may produce HIGH_AIS_PRESSURE but it may not produce AIS eviction, lifecycle demotion, visibility deletion, or priority downgrade."

Critical constitutional protection.

5. Synthetic Pressure Rule
SyntheticPressure may describe background density but may not delete synthetic vessels, block candidates, or mutate spawn intervals. Any future feedback requires a separate spec.

6. Wake Pressure Rule
WakeDensity may describe water-memory pressure but may not evict, shorten, or create wake segments. WakeAuthority remains the only wake memory authority.

7. Camera Interface Explicitly Denied
"ContinuityDensity grants no camera authority. Camera systems may not treat densityClass as camera target priority, cinematic pacing trigger, focus lock, hero promotion, or route steering."

Prevents density creep into camera systems.

8. Evaluation Cadence Using Simulation Time
Recommended cadence: global/sector density every 500ms, debug telemetry every 2000ms. Forbidden: Date.now(), performance.now(), renderer frame timing, wall-clock scheduling.

9. Pure Functions
resolveContinuityDensity(), resolveSectorDensity(), resolveClutterPressure() Ń all pure, no mutation.

Required Amendments (Non-Blocking, Clarifications)
Amendment 1: Density Score Weights Ń Missing Justification
Issue: Sector density formula uses weights: vessel (0.45), wake (0.25), synthetic (0.15), continuity load (0.15). No justification for these specific weights.

Fix: Add rationale:

diff
+ ## Density Score Weight Justification
+ 
+ Vessel density is weighted highest (0.45) because vessel presence is the primary
+ contributor to visual and cognitive density in harbor observation.
+ 
+ Wake density (0.25) is secondary Ń wakes persist after vessels pass,
+ contributing to perceived motion memory but not active occupancy.
+ 
+ Synthetic pressure (0.15) reflects background fill, which should influence
+ density but not dominate it (AIS supremacy).
+ 
+ Continuity load (0.15) accounts for low-confidence or stale vessels that
+ still occupy continuity memory without full visual authority.
+ 
+ These weights are advisory defaults. Implementations may adjust with
+ documented justification, but must preserve AIS supremacy.
Amendment 2: Clutter Pressure Weights Ń Missing Justification
Issue: Clutter pressure formula uses sector density (0.55), wake density (0.25), background/ghost ratio (0.20). No justification.

Fix: Add rationale:

diff
+ ## Clutter Pressure Weight Justification
+ 
+ Sector density (0.55) is the dominant factor Ń dense sectors produce clutter.
+ Wake density (0.25) adds water-memory pressure without overwhelming visual input.
+ Background/ghost ratio (0.20) captures low-priority vessel accumulation that
+ may not be dense but still adds cognitive load.
+ 
+ These weights prioritize active vessel density while acknowledging
+ background accumulation and wake memory.
Amendment 3: Continuity Load Formula Ń Missing Clarity on staleOrLowConfidenceRatio
Issue: Formula references staleOrLowConfidenceRatio but no definition of what constitutes "stale" or "low confidence."

Fix: Add definition:

diff
+ ## Stale or Low Confidence Definition
+ 
+ A vessel is considered "stale" or "low confidence" if ANY of:
+ - signalConfidence < 0.4
+ - continuityAlpha < 0.5
+ - lifecycleState is "COASTING" or "DORMANT"
+ 
+ staleOrLowConfidenceRatio = count(staleOrLowConfidenceVessels) / totalVessels
+ 
+ This ratio measures continuity burden Ń vessels that are present in the
+ continuity system but not providing high-confidence telemetry.
Amendment 4: Sector Input Ń Missing maxVessels Source
Issue: Sector input includes budget.maxVessels but no guidance on where this value comes from (PopulationHierarchy? Zone capacity?).

Fix: Add source definition:

diff
+ ## Sector Budget Source
+ 
+ `maxVessels` per sector should be derived from:
+ - PopulationHierarchy zone budgets (if available)
+ - Ecological zone densityRange.max (from 0523C)
+ - Fallback: global maxVessels distributed by sector area
+ 
+ `maxWakeSegments` per sector should be derived from:
+ - WakeAuthority global budget proportionally allocated
+ - Or zone-specific wake capacity if defined
+ 
+ These values are inputs to density calculation, not runtime limits.
+ Exceeding them does not trigger automatic culling.

Amendment 6: globalDensityScore vs sectorDensityScore Ń Relationship Unclear
Issue: Global score is described but no formula for aggregation from sector scores.

Fix: Add aggregation rule:

diff
+ ## Global Density Score Aggregation
+ 
+ globalDensityScore = average(sectorDensityScore across all sectors)
+ 
+ Or weighted by sector area/importance if sector weights are defined.
+ 
+ Simple average is acceptable for v1.0.0.
+ More sophisticated weighting may be added in future versions.
Amendment 7: Evaluation Cadence Ń Missing "May Skip Frames" Guidance
Issue: Cadence recommendations (500ms) but no guidance on whether evaluation can be skipped if no state changes.

Fix: Add optimization note:

diff
+ ## Evaluation Optimization
+ 
+ ContinuityDensity may skip evaluation if no relevant state changes
+ (vessel count, wake count, tier distribution) since last evaluation.
+ 
+ Skipped evaluations must return the previous result (cached) to maintain
+ deterministic behavior for consumers that expect periodic updates.
+ 
+ Maximum staleness should not exceed 2? the recommended cadence (1000ms).
