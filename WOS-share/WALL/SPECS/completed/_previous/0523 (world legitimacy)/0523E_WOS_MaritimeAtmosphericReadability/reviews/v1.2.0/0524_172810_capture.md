# 1. CORE ARCHITECTURAL PRINCIPLE

The `MaritimeAtmosphericReadability` module acts strictly as an **Interpretation Layer component** within the Wall of Sound (WOS) architecture. It receives authoritative simulation data from upstream runtime tracking systems and transforms it into visual readability descriptors. It possesses absolutely no write authority over simulation truth.

- **Runtime Simulation Layer owns system truth** (Instance lifecycles, coordinate arrays, absolute telemetry).
- **Atmosphere Interpretation Layer owns perception tracking** (Visibility evaluation coefficients, tag masking alerts).
- **Marine Renderer Layer owns final image presentation** (Shaders, buffer compilation, pixel rasterization).

### 1.1 Fundamental Dictum
- **Visibility Is Separate From Existence:** The atmosphere may suppress visual visibility down to absolute zero transparency, but it cannot drop, evict, stall, or delete tracking instances. A vessel completely enveloped by heavy industrial fog remains active inside the primary tracking tables.

# 2. RUNTIME AUTHORITY BOUNDARIES

### 2.1 Explicit Domain Ownership
The module maintains total ownership over the generation of:
- Visual clarity values (`readabilityScore`) and visibility classifications.
- Interface marker visibility states and contextual label suppression recommendations.
- Presentation-facing render parameters (alpha multipliers, blur hints, contrast hints).
- Diagnostics string codes explaining active visibility constraints.

### 2.2 Non-Mutation Enforcements
The module is strictly read-only. It **MAY NOT** write to variables, arrays, or states owned by:
- Automated identifier arrays, velocities, headings, or tracking slots inside `AISRuntime`.
- Budget constraints, focus tiers, or registration rules inside `MaritimePopulationHierarchy`.
- Profile values, physics bounds, or geometric models inside `MaritimeTaxonomyProfiles`.
- Wave segment tracking tables, generation ticks, or ring buffer limits inside `MaritimeWakeAuthority`.
- Seed intervals, density parameters, or local bounds inside `MaritimeSpawnEcology`.
- Graphics memory pipelines, transform passes, or frame buffers inside `MarineRenderer`.
- Viewport focus locations, target locks, or frame configurations inside the camera tracking arrays.
