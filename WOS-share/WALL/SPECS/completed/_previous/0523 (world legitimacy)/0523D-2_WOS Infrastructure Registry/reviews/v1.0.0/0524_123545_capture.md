Strengths (Preserve)
1. Core Distinction: Administrative Truth vs Runtime Truth
"The registry is administrative truth, not runtime truth. It does not mutate runtime systems, own simulation behavior, override spec content, replace implementation tests, or certify code correctness by itself."

This prevents registry from becoming a backdoor authority system.

2. Clean Stage + Freeze Decision Separation
text
Stage: [REVIEW] / [BUILD]
Freeze Decision: REVIEW / GO / STOP
Separate fields prevent conflating "under review" with "ready to build" or "blocked."

3. Implementation Status Granularity
Eight statuses from NOT_STARTED ? READY_TO_BUILD ? IN_PROGRESS ? BUILT_UNVERIFIED ? BUILT_VERIFIED ? PATCH_REQUIRED ? BLOCKED.

Critical distinction: BUILT_UNVERIFIED (code returned) vs BUILT_VERIFIED (audited).

4. Build Scope + Do-Not-Build Scope
Each spec explicitly declares what it builds and what it does NOT build. Prevents scope creep and mis-assigned authority.

Example from 0523A:

yaml
build_scope:
  - taxonomy profile registry
  - AIS class mapping
  - UNKNOWN fallback
do_not_build:
  - renderer behavior
  - population behavior
  - wake rendering
5. Runtime Authority Registry
Clear ownership matrix:

Runtime Owner	Owns	May Not Own
AISRuntime	AIS truth, vessel state	renderer styling
MaritimeWakeAuthority	wake memory, decay	vessel motion, lifecycle
MarineRenderer	presentation	runtime truth
Prevents authority leakage across system boundaries.

6. Supersession Rules
"A newer version does NOT supersede an older version merely because it adds notes. Supersession must be explicit."

Prevents silent version drift and accidental authority changes.

7. Dependency Rules
"If a parent spec is PATCH_REQUIRED, downstream specs may continue review but may not move to [BUILD] unless the dependency is unaffected."

Clean dependency governance without global blocking.

8. Open Issues Tracking
ISSUE-0523D-001 and ISSUE-0523C-001 demonstrate the pattern. Severity levels: BLOCKING, NON_BLOCKING, WATCH.

Required Amendments (Non-Blocking, Clarifications)
Amendment 1: Missing implementation_status: "VERIFIED" Alternative Wording
Issue: BUILT_VERIFIED implies verification happened, but doesn't specify who verified or what criteria.

Fix: Add verification criteria reference:

diff
+ ## Verification Criteria
+ 
+ `BUILT_VERIFIED` requires:
+ - Code review completed
+ - Authority boundaries preserved (no forbidden mutations)
+ - Deterministic rules obeyed
+ - Debug APIs safe (no production leakage)
+ - Integration test passes (if applicable)
+ 
+ Verification must be recorded with reviewer and date.
Amendment 2: Registry Update Frequency Not Specified
Issue: No guidance on how often registry must be updated relative to spec changes.

Fix: Add update rule:

diff
+ ## Registry Update Cadence
+ 
+ Registry must be updated when:
+ - Any spec advances stage (`[REVIEW]` ? `[BUILD]`)
+ - Any spec changes freeze decision (`REVIEW` ? `GO` or `STOP`)
+ - Implementation status changes (e.g., `BUILT_UNVERIFIED` ? `BUILT_VERIFIED`)
+ - A spec is superseded
+ - A new blocking issue is identified
+ 
+ Registry updates should occur within 1 business day of the underlying change.
+ Stale registry is a deployment block for downstream specs.
Amendment 3: Missing DEPLOYMENT_STATUS or Similar
Issue: Registry tracks implementation status but not whether a built system is deployed to production.

Fix: Add optional deployment tracking (defer to v1.1.0):

diff
+ ## Optional Deployment Tracking (v1.1.0)
+ 
+ Future versions may add:
+ 
+ deploymentStatus:
+   | "NOT_DEPLOYED"
+   | "STAGING"
+   | "PRODUCTION"
+   | "ROLLED_BACK"
+ 
+ This is not required for v1.0.0.
Amendment 4: Registry Record Missing reviewer / verifier Fields
Issue: No attribution for who performed review or verification.

Fix: Add optional fields:

diff
+   lastReviewer: string | null;
+   lastVerifier: string | null;
