## 2. Structural Workspace Layout & Workspace Regions

### 2.1 The Four-Region Interface Grid
The layout must force structural isolation across four key interface scopes to keep analysis routines separate from execution pathways:
- **Projection Stage**: Immersive preview area for testing flat map and 2.5D layered spatial data.
- **Control Surface**: Interactive inputs governing the four-corner projection mode, 5 time-of-day phases, and 7 localized weather types.
- **Governance Surface**: Constant readability panel housing metadata lineage data, file stale flags, and token states.
- **Comparison Surface**: Multi-view pane for checking differing color vectors side-by-side.

### 2.2 Mode Disclaimer Constants & Context String Payloads
The workspace layer must enforce persistent textual overlays across all visual modes. The following raw layout strings must remain unalterable:
- **Fiction Mode Configuration Overlay**:

  - line_3: "Not Geographic Truth"

### 2.3 Non-Collapsible Multi-View Metrics Rules
When splitting the workspace view into separate panes within the Comparison Surface, governance markers cannot be summarized or aggregated into unified panels:
- **Rule**: Metrics must calculate independently per active panel loop.
- **Prohibited UI Behavior**: Merging, averaging, inheriting, or collapsing indicators across distinct active comparison surfaces.

## 3. Targeted Gap Analysis & Hardening Interventions

### 3.1 Hardening Against Rogue UI Intent Escalation (Section 16)
- **Analysis**: Section 16 states that the UI may not independently change a tracking record's state from `review` to `activate`. However, if an input component allows an operator to instantly alter the global intent tag via a single un-gated action, a accidental click could trigger a fail-closed crash on the WOS Intake Adapter.
- **Hardening Intervention**: Any manual user interaction intended to shift the state toggle out of `review` must invoke a dedicated verification flow. This flow reads your out-of-band cryptographic signature first. If no active token path is detected, the selector must return to the `review` stance automatically, displaying a clean info toast without broadcasting a broken payload structure down the pipeline.

### 3.2 Guaranteeing Governance Survival Under Graphic Stress (Section 20)
- **Analysis**: The degradation rules appropriately prioritize dropping decorative overlays, particle arrays, and heavy shaders when client hardware bottlenecks. It states that governance text elements must remain fully operational and visible even during intense frame-rate decay.
- **Hardening Intervention**: The rendering engine must split the layout into separate update loops. The Governance Surface must draw via a simple 2D text canvas layer that operates outside the complex 3D post-processing pipeline, ensuring key state data and disclaimers update reliably even if the 2.5D visual preview drops frames.

### 3.3 Locking Environment Context States (Section 7 & 8)
- **Analysis**: Version 1.0.1 adds the strict requirement that active time-of-day variables and current weather parameters must remain continuously visible across the stage viewports.
- **Hardening Intervention**: The client UI must overlay clear state parameters directly inside the stage viewport bounding box (e.g., `ENV_STATE: [NIGHT / FOG]`). If a replay artifact updates these values via a historical file stream, the layout text indicators must modify their values synchronously, preserving complete trace visibility.
