# 0523D_WOS_MaritimeWakeAuthority_v1.0.0

## Purpose

Define maritime wake authority for WOS harbor systems.

Wake systems provide water memory behind moving vessels without allowing wake rendering to mutate vessel truth, lifecycle state, AIS authority, or continuity state.

This spec governs:

- wake ownership
- wake lifecycle
- wake generation eligibility
- wake decay behavior
- AIS vs synthetic wake provenance
- wake density budgets
- wake event data contracts
- renderer observation boundaries

This spec does NOT govern:

- vessel motion
- AIS truth
- dead reckoning
- synthetic vessel steering
- renderer styling
- shader implementation
- camera behavior
- atmospheric scoring
- audio generation
- gameplay systems

## Core Principles

### Wake Memory Follows Vessel Truth

A wake is a passive trace of vessel motion.

```text
vessel motion creates wake memory
wake memory never creates vessel motion
```

### Runtime Owns Wake Authority

Wake authority belongs to runtime infrastructure.

Renderer systems may observe wake state and render it symbolically.

Renderer systems may not:

- create authoritative wake events
- move wake anchors
- alter vessel state
- extend vessel continuity
- fabricate vessel motion from wake persistence

### Wakes Are Not Lifecycle State

A wake may persist after a vessel has moved.

A wake may never:

- keep a vessel alive
- prevent dormancy
- delay eviction
- resurrect a dormant vessel
- imply AIS continuity after telemetry has expired

### Synthetic Wakes Remain Synthetic

Synthetic vessels may generate wakes only when explicitly allowed.

Synthetic wakes must preserve synthetic provenance.

Synthetic wakes may not impersonate AIS wake history.

### Wake Density Must Be Budgeted

Wake systems must obey:

- population tier budgets
- wake density limits
- maximum segment counts
- decay windows
- spatial culling
- renderer LOD constraints

## Continuity Role

WakeAuthority participates in continuity as visual memory.

It may provide:

- recent vessel path traces
- water disturbance persistence
- wake decay fields
- renderer-readable wake geometry descriptors

It may not provide:

- vessel continuity truth
- lifecycle extension
- dead reckoning authority
- interpolation correction
- vessel survival logic

If a vessel becomes DORMANT:

```text
wake decay may continue
vessel lifecycle does not change
```

If a vessel is evicted:

```text
wake segments may decay naturally if already emitted
no new wake segments may be generated
```

```text

2.5D owns presentation.
```

```text

```

Renderer observes WakeRegistry only.

Renderer does not generate wake truth.

## Wake Eligibility Rules

### HERO

- full wake eligibility
- highest emission cadence
- full wake class interpretation

### MID

- standard wake eligibility
- moderate emission cadence

### BACKGROUND

- reduced wake eligibility
- low emission cadence
- may be suppressed under density pressure

### GHOST

- no wake emission by default
- may receive symbolic atmospheric shimmer only in renderer
- renderer shimmer is not WakeRegistry truth

## AIS vs Synthetic Wake Rules

### AIS Wakes

AIS vessels may generate wakes when:

- vessel is active
- speed is above minimum threshold
- taxonomy wakeClass is not NONE
- population tier permits emission
- wake budget permits emission

AIS wakes use provenance:

```text
AIS_VESSEL
```

### Synthetic Wakes

Synthetic ecology vessels may generate wakes only when:

- synthetic vessel exists in MaritimeContinuityEngine
- provenance is SYNTHETIC_ECOLOGY
- population tier permits emission
- synthetic wake budget permits emission
- synthetic mode allows visible water disturbance

Synthetic wakes must:

- carry synthetic provenance
- obey shorter maximum lifetime
- never impersonate AIS wakes
- decay on vessel eviction

## Dormancy / Eviction Rules

If a vessel becomes DORMANT:

- no new wake segments may be emitted
- existing wake segments may continue decaying
- wake persistence does not keep the vessel active

If a vessel is evicted:

- no new wake segments may be emitted
- existing wake segments may decay until expiry
- WakeRegistry may optionally cull all remaining wake segments for that vessel under pressure

If AIS signal recovers:

- wake emission resumes only from current authoritative vessel position
- no wake bridge may be fabricated across missing telemetry unless MaritimeContinuityEngine emitted valid coasting state

## Wake Intensity Model

```text
wakeIntensity =
wakeAuthority
× wakeWidthFactor
× speedFactor
× populationTierFactor
× continuityConfidence
```

Where:

```text
speedFactor = clamp(speedKts / maxExpectedSpeedKts, 0, 1)
```

Wake intensity is bounded:

```text
0.0 → 1.0
```

Wake intensity may not alter vessel speed or heading.

## Wake Budget Governance

WakeAuthority must maintain:

- global wake segment budget
- per-zone wake budget
- per-vessel wake segment cap
- synthetic wake segment cap
- tier-based emission cadence

Recommended defaults:

```ts
const WAKE_MAX_SEGMENTS_PER_HERO = 120;
const WAKE_MAX_SEGMENTS_PER_MID = 80;
const WAKE_MAX_SEGMENTS_PER_BACKGROUND = 24;
const WAKE_MAX_SEGMENTS_PER_GHOST = 0;

const WAKE_MAX_SYNTHETIC_SEGMENTS_GLOBAL = 1000;
```

Budget overflow behavior:

- suppress new wake segment emission
- do not delete vessel
- do not mutate vessel tier
- emit budget telemetry
- optionally cull oldest wake segments first
