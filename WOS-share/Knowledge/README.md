# WOS Knowledge

## Purpose

The **Knowledge** directory contains the foundational documents that define the architectural, operational, and constitutional understanding of the Wall of Sound (WOS) ecosystem.

Unlike project specifications, patches, reports, or implementation artifacts, the contents of this directory are intentionally curated, stable, and long-lived.

This folder serves as the primary knowledge base for both human contributors and AI assistants.

---

# Design Philosophy

Knowledge documents define **how WOS should be understood**, not **what is currently being built**.

They establish the principles used to evaluate new work rather than documenting individual implementation efforts.

The goal is consistency over time.

---

# Intended Audience

This directory is intended for:

- AI assistants
- Constitutional reviewers
- Project contributors
- Future maintainers
- Documentation authors

Reading these documents should provide sufficient context to understand the core architectural philosophy of WOS before reviewing or authoring specifications.

---

# Directory Structure

## Constitution

Foundational governance for the project.

Examples include:

- constitutional principles
- authority boundaries
- project governance
- immutable architectural rules

These documents should change rarely.

---

## Doctrine

Defines the operating philosophy of WOS.

Examples include:

- Runtime Doctrine
- Interpretation Doctrine
- Continuity Doctrine
- Naming Doctrine
- Authority Doctrine

Doctrine explains *why* systems are designed the way they are.

---

## Operations

Operational guidance describing how work is performed.

Examples include:

- documentation standards
- review procedures
- synchronization instructions
- workflow policies
- delivery standards

Operations improve consistency without changing architectural truth.

---

# What Does NOT Belong Here

The Knowledge directory should not contain:

- active implementation specifications
- feature proposals
- patches
- completion reports
- screenshots
- temporary planning notes
- project-specific build status

Those belong within their respective project directories (for example, `PLAY/` or `WALL/`).

---

# AI Knowledge Base

This directory is intentionally organized so it can serve as the primary knowledge base for AI systems.

When configuring an AI assistant, the contents of this directory should be treated as the highest-priority contextual material.

Specifications, reports, and implementation documents provide project history.

Knowledge provides architectural understanding.

---

# Guiding Principle

Knowledge should remain:

- small
- curated
- authoritative
- stable
- easy to review
- easy to synchronize

As the repository grows, this directory should remain focused on enduring guidance rather than accumulating implementation history.

The objective is not to store everything.

The objective is to preserve the ideas that should remain true regardless of how the implementation evolves.
